# CosmicVoyage 最终开发指南（务实执行版 v4.3 工程冻结）
**Minecraft Java 1.20.1 Forge 47.4.0**
`单人开发` | `单一核心Mod` | `阶段递进式发布`
**冻结日期：2026-05-09**

## 零、核心愿景与设计铁律
### 0.1 一句话定位
> 沉浸式星际航行体验核心 + 克制型宇宙交通基础设施平台。
### 0.2 核心哲学
本体极致克制，浪漫体验优先；联动深度扩展，玩家自选。
**"哇"的构成公式：**
巨物感 + 航行存在感 + 视听仪式感

Text
三者缺一不可。
### 0.3 设计铁律
| # | 原则 | 说明 |
|---|------|------|
| 1 | **飞行绝对优先原则** | 任何新增系统不得削弱巨物逼近感、手动驾驶存在感和巡航沉浸感。 |
| 2 | **本体克制原则** | 本体新增内容必须直接服务于以下之一，否则禁止实现：强化飞行/宇宙尺度体验、提供轻量航行辅助或联动必要接口。 |
| 3 | **地面停留限制原则** | 严禁设计促使玩家长时间停留地面刷资源、经营基地或管理生产线的系统。CosmicVoyage 的浪漫属于 `起飞 → 穿越 → 逼近 → 降落 → 再启航`。 |
| 4 | **联动独立性原则** | 本体在零联动环境下必须构成完整、成立且可重复体验的游戏循环。联动负责扩展深度，绝不承担修复本体可玩性的责任。 |
| 5 | **做减法优先原则** | 引入任何新系统前，必须回答：**"它会让玩家更想飞吗？"** 若否，则删除或转为联动。 |

## 一、技术预研阶段（第 0 阶段，约 2 周）
> ⚠️ 在三阶段正式启动前，必须验证以下核心假设。任何一个挂掉，架构需要重大调整。
| 编号 | 验证目标 | 方法 | 通过标准 | 最低测试配置 |
|------|----------|------|----------|-------------|
| **VT-1** | 1,000,000 格距离渲染无浮点抖动 + 玩家实体精度 | 在太空维度放置标记物体，远距离观测，测量渲染坐标误差和单帧像素抖动；同时观察玩家实体在百万格距离时的移动平滑度，记录坐标误差和抖动情况 | 渲染坐标误差 < 0.1 格，单帧抖动 < 1px；玩家实体移动无明显抖动 | i5-10400, 16GB RAM, GTX 1660 |
| **VT-2** | 异步维度预加载耗时可控 | 在上述配置下，使用带完整地形的出生点区块（非空维度），预加载一个月球维度，测量 WorldGen + 区块加载总耗时 | 记录异步预加载的实际耗时，确认启动预生成方案可弥补差距，确保总加载时间在 90 秒巨物窗口内不可感知 | 同上 |
| **VT-3** | 飞船高速物理稳定性与同步 | 1. 飞船加速至 500 格/秒，测试直线飞行渲染<br>2. 在太空维度放置 1 格厚障碍墙，以 500 格/秒撞击，验证连续碰撞检测有效；障碍墙必须位于新加载区块中（非出生点已加载区块），验证碰撞箱在区块加载后立即可用<br>3. 环绕复杂结构飞行<br>4. 持续飞行 1 分钟，测量客户端-服务端位置误差<br>5. 零重力状态下玩家移动、碰撞测试<br>6. 随机厚度 1-8 格障碍阵列碰撞<br>7. 跨 chunk 边界斜向撞击 | 无穿模、无崩溃、帧率 > 30；1 格厚障碍墙通过（含新加载区块场景）；位置误差 < 0.5 格，无位置回滚；1-8 格障碍阵列无漏检；跨 chunk 边界碰撞正常 | 同上 |
| **VT-4** | 太空维度长距离飞行性能 | 500 格/秒持续飞行 5 分钟，监测内存与帧率 | 内存占用涨幅 < 500MB，帧率波动 < 10 帧 | 同上 |
> **决策点：** 四项全部通过 → 启动第一阶段。任一项失败 → 调整架构后再启动。

### 1.1 Kill Criteria（项目终止红线）
> 🔴 以下为硬性架构死刑条件。任一条件被触发，必须停止当前路线，进行架构级重构或重新评估项目可行性。
| 编号 | 终止条件 | 触发后行动 |
|------|----------|------------|
| **KC-1** | VT-1 抖动不可接受，且确认非 camera-relative 渲染可修复（即根源在玩家实体世界坐标精度） | 评估太空维度坐标原点重置方案；若不可行，将最大飞行距离缩减至 100,000 格以内 |
| **KC-2** | VT-3 核心碰撞/同步问题迫使 Mixin 侵入点超过 25 个，或必须修改原版 Entity living tick 核心循环 | 评估是否可降级为"限制高速区域"方案；若不可行，将飞船速度软上限降至 200 格/秒 |
| **KC-3** | VT-4 内存涨幅 > 1.5GB 且 GC 暂停时间超过 500ms | 评估是否可接受"太空维度仅保留 3×3 区块"的极端限制；若不可行，放弃实时区块加载，改为全静态场景 |
| **KC-4** | 维度切换稳定失败率 > 1%（即每 100 次切换出现 1 次以上崩溃/数据损坏） | TRANSITIONING 状态机判定为不可靠，切换方案降级为"手动确认传送"（无特效过渡） |
> ⏱️ **Kill Criteria 唯一激活窗口：** 第 0 阶段及第一阶段前 4 周。超过此窗口，KC 仅作为警告标记，不再触发项目终止。

### 1.2 第 0 阶段隐性关注点
- **Mixin 实际侵入范围：** 评估关闭服务端位置校验、六自由度移动、摄像机姿态控制是否需要 Mixin；记录 Forge 原生 API 的覆盖边界与无法覆盖的硬点
- **玩家实体在百万格坐标下的浮点精度衰减程度：** 记录坐标误差与抖动幅度，确认是否影响可玩性
- **零重力客户端运动预测冲突程度：** 记录位置回滚频率与幅度，确认是否需要修改客户端运动逻辑
- **CCD 与区块加载的协同延迟：** 记录高速进入新区块时碰撞箱可用延迟，确认最坏情况

## 二、技术架构
### 2.1 三维度核心（Ad Astra 教训）
| 层级 | 内容 | 维度类型 | 数量限制 |
|------|------|----------|----------|
| **Layer 1** | 主世界（地球表面） | 原版 `Level.OVERWORLD` | 固定 |
| **Layer 2** | 太空维度（太阳系模型） | 自定义唯一维度 | 1 |
| **Layer 3** | 星球表面（独立） | 自定义按需加载 | ≤ 6 |
**严格控制：**
- `DimensionType` 注册：太空 1 + 星球表面 ≤ 6
- 同时活跃 `ServerLevel`：≤ 2（太空 + 当前星球）
- 原版地狱/末地：复用不改，不新增
- 过渡状态机：增加 `TRANSITIONING` 状态，过渡期间禁止新维度加载请求，彻底断开旧维度后再切入新维度
---
### 2.2 技术铁律
| # | 铁律 | 说明 |
|---|------|------|
| 1 | **依赖最小化** | 优先 Forge 原生 API，精准使用 Mixin，零第三方库依赖 |
| 2 | **数据安全优先** | 维度保存主线程执行，禁止异步 `save()` |
| 3 | **Mixin 策略** | 追求精准可控而非数量最小化，每个 Mixin 有明确必要性文档、侵入点说明、降级预案 |
| 4 | **内存控制** | Phase 1 仅 unload chunk，活跃维度 ≤ 2；Phase 2 视验证结果允许销毁实例 |
| 5 | **光影兼容** | v1.0 明确声明不支持任何 ShaderPack |
---
### 2.3 维度生命周期
| 操作 | 行为 |
|------|------|
| 玩家进入 | `getLevel()` 获取实例，强制加载出生点区块 |
| 玩家活动 | 区块正常加载/卸载，修改内存数据 |
| 玩家离开 | 主线程安全保存，解除强制加载，释放区块 |
| 空载状态 | `ServerLevel` 映射保留，仅卸载区块内容，**不销毁实例** |
| 空载超时 | Phase 1 **不实现**维度销毁。Phase 2 视验证结果引入彻底销毁机制 |
| 过渡态 | `TRANSITIONING` 标记，旧维度卸载，新维度未激活；活跃维度数始终保持 ≤ 2 |
---
### 2.4 重力与飞船同步
- **重力：** 事件覆写为主，六自由度移动几乎必然需要 Mixin。
    - 降级方案：状态效果实现
- **飞船同步：** 客户端权威，单人优先；联机完全冻结

## 四、飞船与航行系统
### 4.1 核心参数
| 参数 | 数值 | 说明 |
|------|------|------|
| 推进力 | `0.5f` | 适中，持续加速 |
| 转向速度 | `2.0f` | 即时响应 |
| 惯性阻尼 | `0.02f` | 极弱，漂移感 |
| 软上限速度 | **500 格/秒** | > 500 后推力指数衰减 |
| 巡航速度 | **500 格/秒** | Shift+W 长按激活，视觉星空拉伸 |
| 手动飞行速度 | **100 格/秒** | 默认手感参考 |
---
### 4.2 巡航模式
- **触发：** `Shift + W` 长按 3 秒
- **效果：** 速度平滑提升至 500 格/秒，星空拉伸，粒子拖尾，太空维度关闭所有区块加载
- **退出：** 松开 W / 接近星球（< 50,000 格）/ 鼠标大幅移动
- **减速仪式感：** 平滑减速 2-3 秒，星空效果渐退
---
### 4.3 本体设施（Phase 2.5）
> 以下为本体提供的轻量航行辅助设施。每个设施必须回答：**"它让玩家更想飞吗？"**
| 设施 | 功能 | 设计约束 |
|------|------|----------|
| **Orbital Beacon** | 轨道导航信标；接近时提供自动减速提示 | 🚫 不可替代手动驾驶 |
| **Docking Array** | 对接辅助阵列；标准化对接接口 | 只辅助对齐，不自动停靠 |
| **Relay Network** | 中继网络；可视化已建立航线 | 纯视觉标记，无传送功能 |
| **Landing Cache Pad** | 着陆缓存平台；预设的补给/缓存点 | 无生产功能，仅存储 |
> 🎯 **设计目标：** 建立并维护地月/地火间的可靠交通网络。增强重复飞行的**欲望**，而非替代飞行。
---
### 4.4 玩法防偏移铁律
**严禁以下设计：**
1. ❌ 传统科技树配方（A 矿物 → B 零件 → C 火箭）
2. ❌ 换皮资源星球（去某星球只为挖某矿）
3. ❌ 高频手动跑商任务
4. ❌ 地面复杂经营/产线管理
5. ❌ 任何形式的自动化取代手动飞行
   **允许的自动化：** 仅限服务航行的辅助设施（如 Orbital Beacon 的自动减速提示）。

## 五、视距优化与过渡系统
### 5.1 分层渲染
| 距离层级 | 渲染方式 | 精度 | 性能 |
|----------|----------|------|------|
| > 100,000 格 | 天空盒星图标记 | 1 像素 | 极低 |
| 50,000 - 100,000 | Billboard 精灵 | 8×8 ~ 16×16 | 极低 |
| 10,000 - 50,000 | Billboard + 简单光照 | 32×32 ~ 64×64 | 低 |
| 2,000 - 10,000 | 简化球体（无地形） | 64×64 纹理 | 中 |
| < 2,000 | LOD 球体 + 大气着色器 | 128×128 ~ 256×256 | 中 |
| 切换瞬间 | 完整维度加载 | 全精度 | 高（短暂） |
> ⚠️ **渲染管线要求：** 整个链路必须 camera-relative，禁止绝对坐标泄露。
---
### 5.2 过渡状态机（防崩溃 + 兜底回滚）
NORMAL → PRELOADING → READY_TO_ENTER → TRANSITIONING → NORMAL

Text
- 🛡️ **保护：** 禁止维度操作，屏蔽输入，特效遮掩
- ⏱️ **超时回滚：** 5 秒未完成强制回主世界
- ✅ **失败不崩溃：** 清晰提示，安全返回
- 🔄 **异常恢复：** 强退后检测残留，恢复至安全维度
---
### 5.3 卡顿消除四层防御
| 层级 | 手段 | 目标 |
|------|------|------|
| 预生成 | 启动时预生成星球出生点区块 | 世界生成时间为 0 |
| 预加载 | 距离 10,000 格时异步加载 | 区块加载在巨物窗口内完成 |
| 强制保持 | Forge API，延迟卸载 | 内存中可用 |
| 视觉遮掩 | 全屏特效覆盖 | 重建感知零卡顿 |
