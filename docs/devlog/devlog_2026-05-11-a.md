### 12:00 — 返航系统

实现双向维度切换闭环：
- `SpaceData` — 增加月球锚点（`setMoonAnchor` / `getMoonX/Y/Z`），支持地球+月球双锚点
- `LandOnEarthPacket` — 客户端请求着陆地球
- `CosmicVoyagePacketHandler` — 4 个包注册（ShipSync + MoonTransition + LaunchToSpace + LandOnEarth）
- `TransitionHandler` — 太空维度检测距离地球<500 或距离月球<200 触发着陆
- `ShipInputHandler` — 月球维度 y≥300 也触发 `LaunchToSpacePacket`

**编译错误**：`SpaceData` API 改名后 `CvSpaceCommand.java` 仍使用旧方法名（`setAnchor` / `hasAnchor` / `getX/Y/Z`）。修复 5 处调用。

**编译错误**：`PacketHandler` 中 `player.level().dimension()` 返回类型是 `ResourceKey<Level>`，误当作 `Level` 使用。修复。

### 14:00 — 传送到 (0, 0, 0) 问题

从主世界传送到太空后出现在 `(0, 0, 0)` 而非 `(0, 200, 0)`。

**根因**：`dimension_type/space.json` 的 `logical_height=256` + `min_y=-64`，上限为 192，y=200 超出范围被钳制。

修复：`space.json` 和 `moon.json` 的 `logical_height` / `height` 改为 384。

**运行时错误**：`moon.json` dimension_type 缺少 `monster_spawn_block_light_limit` 和 `monster_spawn_light_level` 字段。添加。

### 15:00 — 循环传送 Bug

从主世界到太空后，立刻被传回主世界 y=200。

**根因**：`handleLaunchToSpace` 传送目标 `(0, 200, 0)` 距离地球中心 `(0, 0, 0)` = 200 格，小于 `TransitionHandler.TRIGGER_LAND_EARTH = 500`，一进入太空就触发地球着陆，形成死循环。

修复：传送目标改为 `(0, 200, -600)`，距离地球 ≈ 632 > 500。

### 16:00 — SpaceConstants 重构

用户建议"不要硬编码，用一个基准量比例推导"。

创建 `SpaceConstants.java`（唯一硬编码 `EARTH_RADIUS = 200.0f`，其余通过比例推导），并重构 5 个文件引用：
- `EarthRenderer.java`
- `MoonRenderer.java`
- `TransitionHandler.java`
- `CosmicVoyagePacketHandler.java`
- `NavigationHUDRenderer.java`

**问题**：重构基于虚拟机文件而非用户实际代码，导致常量引用与实际值不一致。

### 17:00 — Barrier 方块与画面抽搐

用户发现：
1. Space 下方有屏障方块 — **这是原有代码中就有的**（`dimension/space.json` 中 `"block": "barrier"`），不是今天引入的
2. 飞向月球时画面抽搐、坐标线性增长但画面横跳

我错误地判断抽搐根因是 `handleShipSync` 的 `hasImpulse = true`，并修改了 ShipSync 逻辑。

### 18:00 — 维度设置修改

用户要求去掉边界限制。

修改 `dimension_type/space.json`：`height: 4064`, `min_y: -2032`, `logical_height: 4064`。

**崩溃**：`dimension/space.json` 删除了 `layers` 字段 → `No key layers` → Flat 生成器硬性要求 `layers` 字段。恢复 `layers`（air 替代 barrier）。

### 19:00 — ShipSync 改坏冻结功能

将 `handleShipSync` 改为"只同步速度，不 setPos"，移除了 `hasImpulse = true`。

**后果**：飞船超前、被甩下、解除骑乘 — **5月10日冻结的"零回跳修复"被破坏**。

紧急恢复 `handleShipSync` 到原始版本（`setPos + hasImpulse`）。

### 20:00 — ShipRenderer 大坐标精度修复（错误方向）

用户报告 5000+ 格时飞船消失、HUD 闪烁、视角锁死。

我在 `ShipRenderer.render()` 中添加了 camera-relative translate（`entity.getX() - camPos.x`），并删除了 `super.render()` 调用。

**后果**：小坐标下（太空出生点 ~600 格）飞船和视角都开始抖动 — **双重偏移**。`LevelRenderer.renderEntity()` 已经做了 camera-relative translate，ShipRenderer 又做了一遍。

### 21:00 — ShipManualRenderer（更严重的错误）

试图在 `RenderLevelStageEvent.AFTER_ENTITIES` 中手动渲染飞船线框，绕过 `LevelRenderer` 的 float 精度限制。

**后果**：所有坐标下都剧烈抖动，视角无法转动，地球渲染也跳变 — **彻底破坏渲染管线**。

## 经验教训

1. **冻结 = 完全不碰**。不只是"不改那个方法"，而是任何相关文件都不碰。
2. **修改前确认现有行为**。打日志、查源码，不要凭假设改代码。
3. **先跑通，再漂亮**。重构是最后一步，不是中间步骤。
4. **一次只改一个文件**。验证通过再改下一个。
5. **用户代码为准**。虚拟机文件 = 垃圾。
6. **连续 3 次修改没解决问题 → 停止，回退**。
