## 四、技术决策记录

### 新决策

| 决策 | 选择 | 理由 |
|:---|:---|:---|
| SpaceConstants 设计 | 唯一源头 `EARTH_RADIUS` | 5月11日常量重构教训：基于虚拟机文件导致引用混乱。本次从实际代码推导，所有值可数学验证 |
| Packet 拆分粒度 | 每个 Packet 独立文件 | 三合一文件 291 行已超过可读阈值，第二阶段新增行星 Packet 会进一步膨胀 |
| NBT 版本兼容 | v1→v2 自动迁移 | 已有玩家存档使用单锚点格式，不能破坏向后兼容 |
| 基类抽取时机 | Phase 2 启动前完成 | Earth/Moon 重复率 74-77%，第二阶段新增火星/金星会重复同样模式 |

### 5月11日事故复盘（写入冻结清单）

| 错误 | 教训 | 当前预防措施 |
|:---|:---|:---|
| E1: 基于虚拟机文件修改 | 用户代码为准 | 整固期所有方案基于实际文件 |
| E2: 修改冻结代码 | 冻结 = 完全不碰 | 11 项冻结决策清单，Phase 2 禁止触碰 |
| E3: 未理解底层机制就修改 | 先查源码再动手 | API 契约中标注了每个决策的技术依据 |
| E4: 过度工程化 | 先跑通再漂亮 | SpaceConstants 只抽一个 EARTH_RADIUS，不做"一整套常量系统" |
| E5: 连续 3 次修改未解决 → 继续加改 | 连续 3 次失败 → 停止回退 | 整固期制定检查清单，逐项验证 |
