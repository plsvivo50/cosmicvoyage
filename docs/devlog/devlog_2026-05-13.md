### 成果 2：AI 辅助编程版权分析完成

- **结论**：原创作者身份完整，MIT 许可证效力无瑕疵
- **依据**：
  - 独创性体现在整体架构、设计基调、功能取舍（思想层）
  - AI 仅承担语法实现与 API 查询（表达工具层）

- **平台条款**：GitHub Copilot / OpenAI / Anthropic 均声明输出归用户所有
- **风险提示**：训练数据污染（AI 生成与他人 GPL 代码相似片段），需警惕"许可证污染"

### 成果 3：工程隐患识别与复盘机制确立

- **预警信号**：AI 对代码质量的客观评价持续走低
- **根因**：
  - 上下文污染（代码库膨胀，AI 被迫在混乱上下文中生成代码）
  - 耦合度隐性上升（后期模块被迫兼容前期隐式契约）
  - 重复造轮子（AI 不知道已有工具方法）
  - 边界侵蚀（层间边界开始出现跨层调用）
- **性质**：正反馈陷阱，必须人工介入刹车

## 三、复盘刹车计划（3 天整固模式）

> **目标**：从"冲刺模式"切换为"整固模式"，停止新增大功能，完成架构测绘、异味大扫除、接口固化。

### Day 1 —— 架构测绘（Architecture Mapping）

- **输入**：`src/main/java` 完整包结构
- **AI 动作**：
  1. 绘制模块划分图（每个包的职责）
  2. 分析包间依赖方向（是否存在循环依赖、反向依赖）
  3. 标记与 Forge / Minecraft 原生类的耦合点
  4. 基于文件结构推测高风险区域（初步异味嗅探）
- **人类决策**：对比"实际架构"与"理想架构"，标记偏离点
- **产出文件**：`docs/architecture/actual_vs_intended.md`

### Day 2 —— 代码异味大扫除（Code Smell Audit）

- **扫描范围**：核心模块（飞行系统 + 维度切换 + 数据包系统）
- **AI 动作**：
  1. 魔法数字 / 硬编码字符串检测
  2. 空 catch 块 / 吞异常检测
  3. 资源泄漏（IO、网络句柄、GL 资源）检测
  4. 重复代码片段对比
  5. 过长方法（>50 行）与类职责单一性检查
  6. 跨层调用标记
- **分级标准**：
  - **P0**：可能导致崩溃、数据丢失、安全漏洞，第二阶段开始前必须修复
  - **P1**：影响可维护性，建议第二阶段初期修复
  - **P2**：风格问题，可分期处理
- **产出文件**：`docs/tech_debt/P0_P1_P2.md`

### Day 3 —— 接口契约固化（API Freezing）

- **目标**：将核心模块的对外接口固定，后续开发只能调用接口，不能穿透实现
- **AI 动作**：
  1. 从核心类（飞行控制器、维度管理器、数据包调度器）提炼最小公共接口
  2. 输出建议的接口签名与 JavaDoc 模板
  3. 评估当前实现与接口的偏离度
- **人类决策**：
  1. 确认接口签名（"这是我对外的承诺"）
  2. 基于 Day 1-2 报告，圈定第二阶段开始前必须修复的 P0 项
  3. 制定重构方案（哪些文件动，哪些冻结）
- **产出文件**：`docs/phase2_prep/required_refactors.md`

## 四、复盘机制（长期嵌入工作流）

### 触发时机

| 触发条件 | 复盘动作 | 耗时 |
|----------|----------|------|
| 每完成 1 个模块封档后 | 模块级复盘 | 20-30 分钟 |
| 每累计 3 个模块封档后 | 跨模块耦合检查 | 40-60 分钟 |
| 每次大版本发布前 | 全项目架构体检 | 1-2 小时 |
| 遇到"诡异 Bug"时 | 定向复盘相关模块 | 15-20 分钟 |

### 模块级复盘清单（每次封档后必做）

1. **接口契约检查**：对外暴露方法是否超出命名职责？
2. **依赖关系图**：是否存在下层调用上层的反向依赖？
3. **代码异味扫描**：魔法数字、空 catch、资源未关闭、NPE 边界
4. **与需求对齐**：是否引入了需求之外的功能？是否过度设计？
5. **测试覆盖直觉**：边界情况是否被处理？

### 跨模块复盘清单（每 3 个模块做一次）

1. **重复代码检测**：是否应抽取公共工具类？
2. **数据流一致性**：模块间数据格式是否匹配？是否存在隐式契约？
3. **错误传播路径**：异常是否会级联？是否有统一错误处理策略？

### AI Prompt 模板（复盘用）

> "请对以下代码进行工程复盘审查。我是项目作者，我的设计意图是：[一句话描述]。请按以下格式输出：
> 1. 职责清单：这个类/包应该做什么，实际做了什么，有没有做多余的事？
> 2. 依赖分析：它依赖了项目内哪些其他模块？有没有不合理的跨层调用？
> 3. 风险标红：列出所有可能的 NPE、资源泄漏、并发问题、魔法数字。
> 4. 简化建议：有没有可以用更简洁方式实现的逻辑？
> 5. 架构契合度：这段代码是否符合我最初的设计基调？有没有偏离？"

### 复盘产出要求（必须落盘）

- 保存路径：`docs/review/YYYY-MM-DD_module_name.md`
- 固定格式：
  - 原始需求（人类写，1 句话）
  - AI 审查摘要（AI 输出，人类贴入）
  - 我的决策（人类写：通过 / 需重构 / 标记技术债）
  - 待办项（如有，同步到开发日志）

## 五、关键原则与心理建设

1. **"看懂代码"≠"背出每一行"**：掌控力 = 知道每个模块的边界、职责和潜在风险
2. **AI 是工具，不是作者**：提供选项的是你，做选择的是你，版权在你
3. **3 天停工是投资，不是浪费**：防止未来 3 周花在调试莫名其妙 Bug 上
4. **里程碑评审是工程管理基本功**：第一阶段完工 = 天然里程碑，必须验地基
